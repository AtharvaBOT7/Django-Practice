Django is a framework for Python.

We use framework to make our work easy, for example if we want to build a login (DB, Permissions, Authentication etc etc) functionality, then there are multitude of ways we can do so but if we choose the traditional way without using the framework, then we will have to write everything from scratch manually therefore we use frameworks like Django to make our life easy as Django already has many in built functionalities which we can reuse.
We might face bugs when we write code manually and therefore it is better to use the pre defined functions within the frameowrks as they are secure and scalable too.

Features of Django:
1. Excellent Documentation
2. Secure and Scalable
3. Fast and Simple
4. SEO Friendly
5. By default admin panel  (CRUD)
6. Extensive library support

In Django we have something called ORM (Object Relational Mapping) which helps us write the relational DB queries and we dont have to manually write everything.

Django uses PBKDF-2 (Password-Based Key Derivation Function 2) algorithm with SHA-256 hashing for encryption.

The Django admin panel which helps to create, update and delete users in the db easily.

## Creating first Django project

Step 1: Run the command below to start a project named mysite.
``` 
django-admin startproject mysite .
```
REMEMBER: mysite is the Project Level File Structure.
This command will create a folder inside your project directory with the name mysite and it will also create a file named manage.py separately outside the mysite folder. Inside the mysite folder, you will see the following files: 
- __init__.py: Helps us to make this folder as an importable package in python, so that we can reuse our code
- asgi.py: (Asynchronous Server Gateway Interface) It is similar to wsgi file but comes with a few additional functionalities
- settings.py: This is the most important file in a Django Project and it is used to add all the configurations of a Django project
- urls.py: It contains all the endpoints which we might have for our website. It is used to provide the address of the resources. It basically tells the application if the user clicks this or wants to visit this resource, then direct them to this site, something like this. 
- wsgi.py: (Web Server Gateway Interface) This is configured with the wsgi server and will help in deploying the application to production servers, it basically has the configurations for how the servers should interact with the web applications.

These are the basic files required to run the Django project, so we can run our Django project now.

Step 2: To run the Django development server, we will run the below command:
```
python manage.py runserver
```
After running this command, it will automatically create a file named db.sqlite3 if everything is successfully configured and db.sqlite3 is a in memory db initialized for our project.

The manage.py is a command line utility which lets us communicate with the Django project in various ways.
The mysite directory is the main project directory where our project files will reside.

### BREAKING DOWN THE Settings.py file:
In this file we will have all the main configurations of our Django project. 

1. BASE_DIR: This is the base directory of our project 
2. SECRET_KEY: This is the key which Django will generate on its own and it is used for hashing.
3. DEBUG: This variable helps us to see the errors while developing but it should be turned off for Production.
4. ALLOWED_HOSTS: This variable helps us for Domain settings. (IP settings)
5. INSTALLED_APPS: These are all the installed apps that Django uses for business logic, any apps we create must be registered here for them to work.
6. MIDDLEWARE: Every request coming into the application and every request going out from the application must pass through all the middleware layers, this is for security reasons.
7. ROOT_URLCONF: This variable tells us that the URLs for this project are all listed out in the mysite/urls.py file.
8. TEMPLATES: Responsible for frontend templates. 
9. WSGI_APPLICATION: This has wsgi server configuration. 
10. DATABASES: By default we are using sqlite3 database but we can edit to our needed database.
11. AUTH_PASSWORD_VALIDATORS: These are the built in password validators which Django comes with by default. Used to check the strength of the user's password.
12. Internationalization: Language code, timezone, it8n: Internationalization, tz: timezone, we can set all of this here.
13. STATIC_URL: Whatever CSS, JS, Images we use on our website are called static files. We need to make special configurations to run the static files.

## How Django Works?
Django follows MVT (Model View Template) software design pattern, in software development, design patterns are way of defining the rules of developing a software application.

MODEL: It is related to each and every database operation. If we want to alter or access the database, we must call the model.
TEMPLATE: It is basically the user facing frontend layout, the user communicates with the database through this frontend. It can be HTML or Bootstrap. It is the only thing that can be seen by the users in a Django application.
VIEW: Acts as a gateway between the Model and the templates (basically tells the app (model) that I need something from the db). We define the logic inside the view. View cannot talk to the DB, for that we must need a model.

In the views we define the URLs using the path function (Example: path('about/), views.about) here about/ is the endpoint and views.about is the function name. It simply means that inside the views module, we have the about function.

### BASIC DJANGO FLOW
A request comes in through the template and first it goes to the urls.py file (this file has all the routes/patterns listed) and from there is routed to the view function appropriately. View has the logic, for example if we want to fetch first and last name, we will tell the model the logic to fetch these items from the db and model will in return fetch this info and return it to the view and view will then return the data to template which will then be displayed to the user.

Now if we run ```python manage.py runserver``` then we will find it running on localhost:8000, and if we try to access the /home slug / endpoint, it will return page not found error. So we will now create our first endpoint. 
Step 1: First we will go to urls.py inside our project folder and register the endpoint "home/" there inside urlpatterns and we will give the function name as views.home and we will then have to import the views package from mysite. path('home/', views.home)

But now we will remove the 'home/' endpoint from the path, so the new path will now be path('', views.home), this means that we will now access the home page as our landing page.

### Loading Static Files in Django
Website generally need to serve additional files like Images, JS, HTML, CSS etc, we refer to those files as static files. 
Suppose we want to server an image and an external css file. So what we do is create the css file and upload the image and then load it inside our html template.

We create the static folder inside the root project directory, in our case we will create it inside "mysite" folder.

Now the next step is to create images and css folders inside the static folder we just created. 
Inside the images folder, we will upload a sample image. 

Inside the syles folder, we will create style.css file and inside that for simplicity we will just list the background colour for now.
After doing that, we will need to link our style.css file to our home.html file. We will just add ```   <link rel="stylesheet" href="css/style.css"> ``` to our html code just above the body before head tag ends.

This will not work as is because Django has a special way of serving the static files. The page will not give an error but it will not change the background to red or include image on our landing page.

So to server the static files, we need to go to settings.py inside our mysite folder and then we add the below code to our settings.py below the static_url variable.
```
STATIC_ROOT = BASE_DIR / 'static'
STATICFILES_DIRS = [
    'mysite/static'
]
```

After this, we will have to load the static file inside our home.html file, we do this by adding the below command at the very top of the html file.
``` {% load static %} ```
and after adding this, we need to change the link to our styles sheet which will be as follows:
``` <link rel="stylesheet" href="{% static 'css/style.css' %}"> ```

We set the static root variable inside the settings.py file to load all the static files for deployment purpose.

### Django Apps 
Django app is a python package specifically created to be used inside a Django project. It is a sub module of the project. Apps are used to keep the code separate and we can have multiple apps in one project.

To create an app, we have to run this command ``` python manage.py startapp <app_name> ```
I created an app named employees and after hitting enter, a new folder named employees is created outside of my project root directory. Inside that folder, we get the same initialized files which we got for the mysite folder. 

Suppose we have different functionalities like login, ordering, support, chatbot, etc etc, then if there were no apps, we would have to maintain all the code under the same project root directory but using apps, we can keep all the functionalities separately so that it is easier for us to debug. Therefore we will create different apps for different features so that it will be easier for us to debug.

Now that we have created the app named employees, it will create the following files by default:
1. __init__.py: This tells the django project that this folder is a python package and can be imported anywhere inside the codebase.
2. admin.py: It is used to display the DB data / model data inside the Django admin panel. For CRUD operations.
3. apps.py: Used to app level configurations to the package.
4. models.py: It contains the essentials fields and data that we are storing in the DB.
5. tests.py: It is used to run any tests in the Django project.
6. views.py: In this file we create business logic for that specific app.

When we run any Django server, we will get this warning:

to solve this error we run ``` python manage.py migrate ```. Django by default gives us some apps like admin, auth, contenttypes, sessions etc and these tables need to be created inside the database before us creating our own data and therefore by running the migrate command, we will have created those default database tables.

We have the database by default and now we need to create tables and fields. So now inside our employees folder, we will navigate to models.py and there we will create our tables and fields.
We create the table using the below command, inside employees/models.py 
```
from django.db import models

class Employee(models.model):
    first_name = models.CharField(max_length=50)
    last_name = models.CharField(max_length=50)
    photo = models.ImageField(upload_to='images')
    designation = models.CharField(max_length=100)
    email_address = models.EmailField(max_length=100, unique=True)
    phone_number = models.CharField(max_length=15, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
```
In this, we have photos and similar to the special settings for the static files, we have special settings for the image upload files.

Once we have created the model, we need to migrate that. The first thing after creating an app is to add that appname in settings.py inside the main django project root folder, in our case it is mysite/settings.py file.

### Migrations
To create the tables in our Database, we need to run 2 commands:
```
python manage.py makemigrations
python manage.py migrate
```
After running makemigrations command, we see: 
```
Migrations for 'employees':
  employees/migrations/0001_initial.py
    + Create model Employee
```
So what makemigrations will do is look for any modifications in the models.py file and then create changes to the migrations file, it created 0001_initial.py file. The tables are not yet created, we have just generated the SQL commands for now.

### Fetching data from Database in Django
Now that we have creted 2 demo records, we will see how do we actually fetch the data from database.

We want the data to be printed on the homepage therefore we will navigate to mysite/views.py because we have configured our homepage there. 

For that we will update the home function to this:
```
from employees.models import Employee

def home(request):
    employees = Employee.objects.all()
    print(employees)
    return render(request, 'home.html')
```
After executing that change, we will see this output in the terminal
<QuerySet [<Employee: Sabo Onepiece>, <Employee: Ace OnePiece>]>

But we dont want the data on the terminal we want it on the frontend, therefore we will need to send the employees vairable results to home.html because that is displaying our frontend.
For that we will have to update our home function again:
```
def home(request):
    employees = Employee.objects.all()
    context = {
        'employees': employees,
    }
    return render(request, 'home.html', context)
```
And we will add {{ employees }} to the home.html file, then we will see the employees displayed on the frontend.

We are now using template tags inside the home.html file
```
{% for i in employees %}
  <h4>{{ i }}</h4>
{% endfor %}
```
We use the template tags when we want to use some logic inside the html file.
